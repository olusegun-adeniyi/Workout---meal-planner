# ARCHITECTURE.md

The structural contract for Forge. Read this before writing or modifying any code. If a decision here conflicts with what feels natural, follow this document — it was set deliberately.

## Stack

- Next.js 15 (App Router) + TypeScript strict
- Tailwind v4 + shadcn/ui
- Neon Postgres + Drizzle ORM
- Anthropic SDK (Claude Sonnet 4.5)
- Web Push + Vercel Cron + Resend
- Deployed on Vercel

## Naming conventions

- Files: kebab-case (`meal-timeline.tsx`)
- Components: PascalCase exports (`MealTimeline`)
- Hooks: camelCase, `use` prefix (`useTodayPlan`)
- API routes: REST verbs in `route.ts`, never `pages/api`-style
- DB columns: snake_case in Postgres, camelCase in TS via Drizzle
- Types: PascalCase, suffixed (`PlannedMeal`, `MealLogInput`)
- Zod schemas: PascalCase + `Schema` suffix (`MealPlanSchema`)

## Data flow rules

- **All DB access lives in `lib/db/queries/`.** Components never import Drizzle directly. Route handlers call query functions.
- **All AI calls live in `lib/ai/`.** Route handlers call orchestrator functions (`generateMealPlan`), never the SDK directly.
- **All time math goes through `lib/time/london.ts`.** No `new Date()` in components. Server is UTC, display is Europe/London.
- **All targets (TDEE, macros) computed by `lib/targets/calculate.ts`.** Never hardcoded in components.
- **Validation:** Zod at every boundary — API inputs, AI outputs, form inputs. No untyped JSON crosses a boundary.

## Component rules

- Server components by default. `'use client'` only when interactive (forms, push subscription, charts).
- Forms use `react-hook-form` + `zodResolver`. No naked `<form>` state.
- Charts: Recharts. Keep it boring, no exotic libraries.
- shadcn primitives in `components/ui/`. Don't modify them after install — wrap and extend instead.

## Styling rules

- Tailwind v4 with CSS variables for tokens. Define palette + spacing + radii in `globals.css` `@theme` block.
- Fonts: Geist Sans (body), Geist Mono (numbers — macros, weights, reps). Numbers in mono looks intentional, not decorative.
- Aesthetic anchor: Linear's restraint, Notion's warmth. Off-white background (`#fafaf9`), near-black text (`#0a0a0a`), single accent (warm coral `#ff5a36` for action — "eat now" buttons, streak flames, progress fills).
- No drop shadows on cards. 1px borders in `rgba(0,0,0,0.06)`. Hover lifts the border opacity to `0.12`.
- Spacing: 4px base unit, but most layouts use `gap-6` (24px) or `gap-8` (32px). Generous, not cramped.
- Motion: subtle. `transition-all duration-150 ease-out` is the default. The only "delightful" animation is the streak counter ticking up on a logged meal — a single confetti burst, restrained.

## Push notification flow

1. User visits site → `<PushPrompt />` renders inline if not subscribed
2. User clicks "Enable" → browser permission prompt
3. Service worker `public/sw.js` registers, generates `PushSubscription`
4. POST `/api/notifications/subscribe` saves subscription to DB
5. Vercel Cron hits `/api/cron/meal-reminder` every 15 min
6. Route queries planned meals + meal logs, computes who needs nudging
7. For each nudge: `lib/notifications/push.ts` sends via web-push library
8. Service worker `notificationclick` handler opens app to `/`

## AI plan generation flow

1. Sunday 6pm: cron fires push "Time to plan next week"
2. User opens `/plan` → if no plan exists for next week, button "Generate plan"
3. POST `/api/plan/generate` → calls `generateMealPlan()` orchestrator
4. Orchestrator: loads profile, loads last week's logs, builds prompt, calls Anthropic SDK
5. Response validated against `MealPlanSchema` (Zod). If invalid, retry once.
6. Validated plan written to `meal_plan` + `planned_meal` tables in a transaction
7. Workout plan generated in parallel with similar flow
8. User redirected to `/plan` to review, swap, confirm

## Auth (v1)

Single-secret gate. `.env`: `APP_SECRET=<random-32-chars>`. Middleware checks a cookie `forge_session` matching the secret. Login page is a single password input that sets the cookie. No user table, no sessions table. Replace with Clerk if I ever invite anyone.

## Build order (mirrors PRD section 11)

**Phase 1 — Foundation (days 1–3)**
1. `npx create-next-app@latest forge --typescript --tailwind --app --no-src-dir`
2. Install: `drizzle-orm`, `drizzle-kit`, `@neondatabase/serverless`, `zod`, shadcn-ui init
3. Build `lib/db/schema.ts`, run `drizzle-kit generate` + `migrate`
4. Build `lib/targets/calculate.ts` with unit tests for TDEE math
5. Write `config/profile.json` with my real numbers
6. Build `lib/db/seed.ts`, run it
7. Build app shell: `app/layout.tsx`, `top-nav.tsx`, `globals.css` tokens, font setup
8. Build Today screen reading seeded data — static, no logging yet
9. Deploy to Vercel, confirm it loads

**Phase 2 — Logging (days 4–6)**
10. POST `/api/meal-log` + `lib/db/queries/meals.ts`
11. Today screen interactive: [Ate it] [Swap] [Skip] buttons with optimistic UI
12. Workout logging: exercise rows, set logging
13. Weight logging widget on Today screen
14. Custom meal form (text only, no photo yet)

**Phase 3 — AI + push (days 7–9)**
15. `lib/ai/client.ts`, `prompts.ts`, `schemas.ts`
16. `generateMealPlan()` orchestrator + `/api/plan/generate`
17. `/plan` page: week grid + swap dialog + confirm flow
18. Workout plan generation parallel
19. Service worker, VAPID setup, `<PushPrompt />`, `/api/notifications/subscribe`
20. Cron routes: meal-reminder, morning-briefing, sunday-plan
21. Resend email integration for morning briefing
22. `estimateMacros()` with photo + text, integrate into custom meal form

**Phase 4 — Polish + ship (days 10–14)**
23. Progress page charts (weight trend, adherence, volume)
24. Photo upload via Vercel Blob
25. PWA manifest, icons, install prompt
26. Empty states, loading states, error boundaries
27. Deploy, install on phone home screen, dogfood 48h
28. Triage and fix what breaks

## What good looks like

- Every page renders in < 200ms server-side
- Every POST returns in < 300ms (AI calls excepted)
- Logging a planned meal takes one tap, no confirmation dialog
- Push notification arrives within 60s of scheduled time
- Lighthouse score: Performance 95+, Accessibility 100, PWA installable
- I open the app first thing in the morning and last thing at night, voluntarily, for 7 days running

## What "drifting from the plan" looks like (don't do this)

- Adding a settings page because "it'd be easy"
- Building a recipe library before AI generation works
- Using a CSS-in-JS library instead of Tailwind
- Reaching for tRPC, GraphQL, or any RPC layer — REST routes are fine for one user
- Adding Sentry, PostHog, or any analytics — I don't need to track myself
- Adding a marketing landing page — there's no marketing
- Migrating to Prisma — Drizzle is the choice
- Building a "share my progress" button — single-user, no audience
